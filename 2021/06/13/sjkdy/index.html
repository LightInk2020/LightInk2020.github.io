case

  • sql中 列名、nvl(sal,0)、(select count(*) from emp)、case when end 等返回一个值的都是表达式，然后都可以出现在case中

  • case作用：类似C中switch 用来分支判断

  • select语句作用：

    • select h1,h2 逐行运算表达式 通过对输入的表进行逐行运算
      • 输入多少行，就输出多少行
      • select 1 from emp 输出emp的行数个值为1的行

  • case的两种形式

    • case 列名 when 标量 then …end
    • case when 表达式 then …end

相关子查询

  • 判断是否相关？

    • 子查询是否可以独立运行
    • 嵌套的子查询是否用到了外部查询语句中的表或重命名的表

  • 相关子查询本质：对外部表的每一行依此进行执行select子查询

    • 类似双重循环，只不过内部循环使用外部循环的变量

    • 当外层查询执行到子查询位置时，子查询已经进行完成并将结果（标量）返回给外查询

范式

  • 数据依赖

    • 单值依赖（函数依赖）：当t1(α)=t2(α)时，总有t1(β)=t2(β)
    • α决定β 或称 β依赖α

  • 函数依赖类型

    • 平凡/非平凡

      • 平凡 β包含于α时，α→β 例如：AB→A

      • 非平凡的 例如：AB→BC A→B

        • 部分函数依赖

        (sno,cno)→sname 就是部分函数依赖，因为存在sno→sname

        为什么不直接用(sno)→sname？ 因为(sno,cno)是关系模式的码，一般找与码有关的函数依赖

        • 完全函数依赖

        sno→sname

        • 传递函数依赖

        sno→dno→dname

  • 2NF 消除了非主属性对于码的部分函数依赖

  • 3NF 消除了非主属性对于码的传递函数依赖

    消除了非主属性对于码的传递函数依赖就已经消除了非主属性对于码的部分函数依赖

  • BCNF 消除了主属性对于码的部分函数依赖、传递函数依赖

  • 如何确定码？

    • 能通过函数依赖确定所有属性的，就是码

      如R(A,B,C,D,E)，F{A→B; C→DE; AB→CE}找出所有的候选码。

      解：

      先写出F+{A→B; C→DE; AB→CE; A→ABCDE}

      所有只在函数依赖右边出现，而不在左边出现的，都不可能是候选码，所以只考虑A,B,C

      只知道A可以推出所有ABCDE，即A+=R，所以A→R，A是候选码

      而B、C不可以，所以B、C不是候选码

  • 判断是否满足BCNF —即是否都满足两个条件之一

    • 平凡函数依赖
    • 函数依赖的决定因素是否是超码

接着上面一题，F+{A→B; C→DE; AB→CE; A→ABCDE} 主码是A

先找出所有左边不是超码的函数依赖，这里面是C→DE，即是由于C→DE导致不满足BCNF

就将他单独拿出来作为一个关系模式R2(CDE)，原关系模式去掉C→DE中右边存在而不在左边存在的属性即DE作为另一个关系模式R1(ABC)。

即实现R的BCNF分解，R分解成两个BC范式R1和R2。

即r(R)由于α→β导致不满足BCNF，就拆分成r1(R-(β-α))和r2(α∪β)

  • 分解（包括BCNF）都要是无损分解

无损：例如r拆分成r1和r2，r1 natural join r2 要和原来的r的数据一样
